# Baanbaan Setup API — Specification for COSA Integration

**Version:** 1.0
**Date:** 2026-03-29
**Status:** Specification — required before COSA Phase 1 can be set up without manual config

---

## Overview

The COSA setup wizard (`npm run setup`) discovers and configures itself against a Baanbaan instance by calling a set of lightweight HTTP endpoints on Baanbaan's existing API server. These endpoints are only used once — during initial setup — and are disabled afterward.

This document defines the three endpoints Baanbaan must implement, the PIN-based security model, and the exact response shapes the COSA wizard expects.

---

## Security Model

### Local-network only

The setup endpoints must only be reachable on the local LAN. They must not be exposed through the Cloudflare tunnel. Baanbaan should bind these routes to the local interface only, or check that the request source IP is in a private range (`10.x`, `172.16–31.x`, `192.168.x`).

### Setup PIN

`POST /setup/register-ssh-key` (the only write endpoint) is protected by a 6-digit numeric PIN. The PIN:

- Is generated by Baanbaan using `crypto.randomInt(100000, 999999)`
- Is valid for **24 hours** from generation
- Is **single-use** — invalidated immediately after a successful registration
- Is displayed on the Baanbaan device screen and/or included in the initial setup email sent to the operator when Baanbaan is first installed
- Can be regenerated on demand via a physical button press or a CLI command on the Baanbaan Pi (`baanbaan generate-setup-pin`)

**Rationale:** The PIN is only needed once. An attacker would need to be on the same LAN and know the PIN within 24 hours — an acceptable risk for a local network setup flow.

### Setup mode

Baanbaan starts in setup mode when:
1. It has never been set up before (no `data/cosa_registered` flag file), OR
2. The operator explicitly triggers setup mode via `baanbaan generate-setup-pin`

After a successful `POST /setup/register-ssh-key`, Baanbaan writes a `data/cosa_registered` flag file. Subsequent calls to `register-ssh-key` with a valid PIN will return `409 Conflict` (already registered) rather than registering a new key, unless the operator explicitly runs `baanbaan reset-cosa` to clear the flag.

The `GET /setup/info` endpoint remains available at all times (it returns only non-sensitive read-only data).

## Endpoints

### `GET /setup/info`

Returns the information COSA needs to generate its configuration files. No authentication required.

**Available:** Always (setup mode and production mode).

**Response `200 OK`:**

```json
{
  "appliance": {
    "name": "Baanbaan POS",
    "version": "1.4.2",
    "runtime": "Bun",
    "os": "Raspberry Pi OS (Bookworm)",
    "deploy_path": "/home/baanbaan/app",
    "api_port": 3000,
    "timezone": "America/New_York",
    "pos_adapter": "Clover"
  },
  "network": {
    "lan_ip": "192.168.1.10",
    "mdns_hostname": "baanbaan.local"
  },
  "ssh": {
    "user": "baanbaan",
    "host_key_fingerprint": "SHA256:AbCdEfGhIjKlMnOpQrStUvWxYz0123456789+/AAAA="
  },
  "database": {
    "path": "/home/baanbaan/app/data/baanbaan.db"
  },
  "process_supervisor": {
    "type": "systemd",
    "service_name": "baanbaan"
  },
  "setup": {
    "cosa_registered": false,
    "pin_expires_at": "2026-03-30T14:00:00.000Z"
  }
}
```

**Field notes:**

| Field | Notes |
|---|---|
| `appliance.name` | Human-readable display name (shown to operator during setup) |
| `appliance.version` | Semver string of the Baanbaan app |
| `network.lan_ip` | The LAN IP Baanbaan is currently listening on. Used by COSA as the stable address in config (not the mDNS hostname, which may not resolve on all networks). |
| `ssh.host_key_fingerprint` | SHA-256 fingerprint in OpenSSH format: `SHA256:...`. Obtain with: `ssh-keyscan -t ed25519 localhost \| ssh-keygen -l -E sha256 -f -` |
| `ssh.user` | OS user COSA will SSH in as. This user needs read access to the database and `systemctl show` on the service unit. |
| `database.path` | Absolute path on the Baanbaan Pi. |
| `setup.cosa_registered` | `true` if COSA has already registered a key. The wizard uses this to skip the PIN step with a friendly message if setup is being re-run. |
| `setup.pin_expires_at` | ISO 8601 timestamp of when the current PIN expires. `null` if no PIN has been generated or setup mode is inactive. |

### `POST /setup/register-ssh-key`

Registers the COSA Pi's SSH public key in the `baanbaan` user's `~/.ssh/authorized_keys`. After this call succeeds, COSA can connect to Baanbaan via SSH without a password.

**Available:** Setup mode only (returns `503 Service Unavailable` in production mode unless `baanbaan generate-setup-pin` was run).

**Request body:**

```json
{
  "public_key": "ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAA... cosa@cosa-pi",
  "pin": "847291"
}
```

| Field | Type | Notes |
|---|---|---|
| `public_key` | string | Full contents of the COSA Pi's `~/.ssh/id_ed25519_cosa.pub` |
| `pin` | string | 6-digit PIN displayed on the Baanbaan device |

**Responses:**

| Status | Meaning | Body |
|---|---|---|
| `200 OK` | Key registered successfully. COSA can now SSH in. | `{ "ok": true, "message": "COSA registered successfully." }` |
| `401 Unauthorized` | Incorrect PIN. | `{ "error": "incorrect_pin", "message": "Incorrect PIN." }` |
| `410 Gone` | PIN has expired (>24 hours). | `{ "error": "pin_expired", "message": "Setup PIN has expired. Generate a new one on the Baanbaan device." }` |
| `409 Conflict` | COSA is already registered. | `{ "error": "already_registered", "message": "A COSA key is already registered. Run baanbaan reset-cosa to start over." }` |
| `503 Service Unavailable` | Setup mode is inactive. | `{ "error": "setup_inactive", "message": "Baanbaan is not in setup mode. Run baanbaan generate-setup-pin to activate." }` |
| `400 Bad Request` | Missing or malformed fields. | `{ "error": "invalid_request", "message": "..." }` |

**Implementation notes:**

On a `200` response, Baanbaan must:
1. Append the `public_key` to `/home/baanbaan/.ssh/authorized_keys` (create the file and `.ssh` directory if they don't exist, with permissions `700` on the dir and `600` on the file)
2. Invalidate the PIN so it cannot be reused
3. Write the `data/cosa_registered` flag file
4. Log the registration event (timestamp, source IP)

---

### `GET /setup/status`

Lightweight liveness check used by the setup wizard to confirm the connection before proceeding. Returns the current setup state.

**Available:** Always.

**Response `200 OK`:**

```json
{
  "ok": true,
  "cosa_registered": false,
  "setup_mode_active": true
}
```

## PIN Display and Generation

### On first boot

When Baanbaan starts for the first time (no `data/cosa_registered` flag), it should:

1. Generate a 6-digit PIN
2. Display it prominently on the device screen: `COSA Setup PIN: 847291 (expires in 24h)`
3. Include it in the initial welcome email sent to the operator
4. Log it to stdout: `[setup] COSA setup PIN: 847291 (expires 2026-03-30T14:00:00Z)`

### On demand

The operator or installer can generate a fresh PIN at any time:

```bash
# On the Baanbaan Pi
baanbaan generate-setup-pin
# → COSA Setup PIN: 391847 (expires in 24h)
```

This invalidates any existing PIN and generates a new one, restoring setup mode.

### Reset (re-run setup)

To allow COSA to re-register (e.g. replacing the COSA Pi):

```bash
baanbaan reset-cosa
# → COSA registration cleared. Run 'baanbaan generate-setup-pin' to issue a new PIN.
```

This removes the `data/cosa_registered` flag and removes the existing COSA key from `authorized_keys`. The operator then runs `baanbaan generate-setup-pin` and re-runs `npm run setup` on the COSA Pi.

## Required System State for SSH Access

After `POST /setup/register-ssh-key` succeeds, COSA connects via:

```
ssh -i ~/.ssh/id_ed25519_cosa baanbaan@192.168.1.10
```

For this to work, the `baanbaan` OS user must be able to:

| Action | Requirement |
|---|---|
| SSH login with key | Key in `~/.ssh/authorized_keys`; SSH daemon accepting key auth |
| Run `sqlite3` against the database | Database file readable by `baanbaan` user (`chmod 640`, owner `baanbaan:baanbaan`) |
| Run `systemctl show baanbaan` | No special permissions needed — systemd property reads are unprivileged |
| Run `systemctl show baanbaan --property=...` | Same as above |

COSA does **not** require `sudo` for any Phase 1 operation. Do not grant it.

## mDNS / Bonjour (Optional but Recommended)

If Baanbaan enables mDNS via `avahi-daemon` (Linux) and advertises itself as `baanbaan.local`, the setup wizard will find it automatically without the operator needing to look up an IP address.

Recommended Avahi service file (`/etc/avahi/services/baanbaan.service`):

```xml
<?xml version="1.0" standalone='no'?>
<!DOCTYPE service-group SYSTEM "avahi-service.dtd">
<service-group>
  <name replace-wildcards="yes">Baanbaan POS (%h)</name>
  <service>
    <type>_http._tcp</type>
    <port>3000</port>
  </service>
</service-group>
```

This allows `curl http://baanbaan.local:3000/setup/info` to work from any device on the LAN.

## Summary of Changes Required in Baanbaan

| Area | Change |
|---|---|
| HTTP routes | Add `GET /setup/info`, `POST /setup/register-ssh-key`, `GET /setup/status` |
| Startup | Generate PIN on first boot; display on screen and include in welcome email |
| CLI | Add `baanbaan generate-setup-pin` and `baanbaan reset-cosa` commands |
| File system | Write `data/cosa_registered` flag after successful registration |
| SSH | Ensure `baanbaan` user's `authorized_keys` file is managed correctly |
| mDNS (optional) | Configure `avahi-daemon` to advertise `baanbaan.local` on port 3000 |
| Security | Setup endpoints LAN-only; not exposed through Cloudflare tunnel |
